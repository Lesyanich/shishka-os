# Решение: Создание Глобального Проектного Устава

**Дата:** 2026-03-07
**Агент:** Lead Backend Developer
**Статус:** ✅ Принято и реализовано

## Контекст

Необходимо было создать единый центральный файл правил (`gemini.md`) для управления мультиагентной разработкой в рамках проекта Shishka Healthy Kitchen.

## Принятые Решения

### 1. Централизованный устав в `gemini.md`
- **Решение:** Создать `/gemini.md` как P0 (наивысший приоритет) файл правил
- **Обоснование:** Все агенты (Architect, Chef, Finance) должны работать по единым правилам. Antigravity Kit требует наличия `GEMINI.md` в корне проекта.

### 2. UUID как обязательный тип для всех ID из SYRVE
- **Решение:** PostgreSQL тип `uuid` обязателен для всех ID, пришедших из SYRVE
- **Альтернативы рассмотрены:** `VARCHAR(36)`, `TEXT` — отклонены (нет проверки формата)
- **Обоснование:** UUID обеспечивает целостность на уровне базы данных без дополнительной валидации в коде.

### 3. Мягкие удаления (Soft Deletes) для SYRVE-данных
- **Решение:** Флаг `is_deleted: boolean`. Хранить удалённые записи для аудита и истории.
- **Обоснование:** SYRVE может переиспользовать или восстанавливать ID продуктов. Удалять их из Supabase опасно.

### 4. Слияние источников контекста
- **Рассмотрено:** `Shishka Core Hub.md` (упомянут в задаче), `.agent/rules/GEMINI.md` (найден)
- **Решение:** `Shishka Core Hub.md` не существует как файл на диске (Google Docs stub). Контекст объединён из `.agent/rules/GEMINI.md` + правил пользователя.

## Последствия

- Любой новый агент, читающий `gemini.md`, получит полный контекст: системы, правила данных, структуру папок, протокол документации.
- `HANDOVER.md` создан для передачи эстафеты Architect-агенту.
