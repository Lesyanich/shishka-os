## 🚨 Критические Нарушения (vs. `gemini.md`)

### Нарушение 1: `equipment.id` — TEXT вместо UUID
```
КОНСТИТУЦИЯ (P0): "ALL IDs sourced from SYRVE must be stored as UUID"
ФАКТ: equipment.id = TEXT (например, 'oven_1', 'fermentation_tank')
РИСК: Невозможна прямая синхронизация с SYRVE
```

### Нарушение 2: `daily_plan.id` — INTEGER вместо UUID
```
КОНСТИТУЦИЯ (P0): UUID для всех первичных ключей
ФАКТ: daily_plan.id = SERIAL INTEGER
РИСК: Нет UUID-совместимости для внешних систем
```

### Нарушение 3: `products.code` — TEXT как PK вместо UUID
```
КОНСТИТУЦИЯ (P0): "Product IDs must map 1:1 to SYRVE UUIDs"
ФАКТ: products.code = TEXT ('SOUP-01'), syrve_id = UUID (nullable)
РИСК: SYRVE UUID не является PK — нарушает единый источник правды
```

## 📝 Вывод и Рекомендация

Текущая схема **функциональна** для MVP KDS на Telegram, но **не соответствует** правилам `gemini.md` и не готова к SYRVE-синхронизации.

**Рекомендуемые действия:**
1. Не удалять существующие данные — провести миграцию
2. Добавить `syrve_uuid` / переключить PK на UUID с сохранением `code` как уникального поля
3. Расширить схему новыми таблицами: `nomenclature`, `product_categories`, `maintenance`, `capex_assets`

> Источники CSV (`Production_Flow.csv`, `Resource_Capacity.csv`, `Capex.csv`) не обнаружены на диске — они хранятся как Google Sheets (`.gsheet` стабы) и недоступны для прямого чтения.
