## Risk Assessment

| Risk | Severity | Description |
|------|----------|-------------|
| MCP cross-domain writes | MEDIUM | mcp-finance can write to `products`, mcp-chef can write to `receipt_inbox` |
| No audit trail | LOW | All writes look the same — no way to trace which MCP made a change |
| Admin panel full access | OK | Expected for now — admin panel is the primary UI |

## Recommendation: Phase D Migration Plan

### Step 1: Create per-MCP Supabase service accounts
- `mcp_chef_role` with grants on chef-owned tables
- `mcp_finance_role` with grants on finance-owned tables  
- `mcp_mc_role` with grants on mission-control-owned tables

### Step 2: Replace `USING (true)` with role-based checks
```sql
-- Example: products table
CREATE POLICY products_chef_full ON products 
  FOR ALL TO mcp_chef_role USING (true) WITH CHECK (true);

CREATE POLICY products_finance_read ON products 
  FOR SELECT TO mcp_finance_role USING (true);

CREATE POLICY products_admin_full ON products 
  FOR ALL TO authenticated USING (true) WITH CHECK (true);
```

### Step 3: Issue per-MCP API keys
Each MCP server env gets a unique key that maps to its role.

### Prerequisites (before Phase D)
1. Stable table ownership (see `docs/domain/db-contracts.md`)
2. `supabase db push --dry-run` canary in pre-commit (Phase D6)
3. Test coverage for MCP tools (verify no cross-domain writes in tests)

### Estimated effort
2-3 migrations + key rotation in MCP server configs. ~1 session.
