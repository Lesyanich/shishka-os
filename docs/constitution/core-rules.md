## RULE-SUPABASE-SSOT

Supabase (PostgreSQL 17.6) is the **only source of truth** for the apps. UI is just a mirror. Never invent state in the frontend that doesn't exist in the database.

> Origin: Foundational. Replaces legacy `P0 Rule #1`.

---

## RULE-UUID-COMPLIANCE

All database relationships **must** use UUIDs. No integer FKs. All IDs sourced from SYRVE are stored as `UUID`. No exceptions.

> Origin: Foundational. Replaces legacy `P0 Rule #2`.

---

## RULE-LEGO-ARCHITECTURE

Menu modules follow a strict 4-tier pyramid: `RAW` (raw materials) → `PF` (semi-finished) → `MOD` (toppings/modifiers) → `SALE` (final dish). Every BOM line must reference items from a tier strictly below the parent.

> Origin: Foundational. Replaces legacy `P0 Rule #3`. Detailed conventions in `engineering-rules.md` (RULE-BOM-PREFIX-FILTER).

---

## RULE-NO-DIRECT-DB-EDITS

All DB schema changes are SQL migrations in `services/supabase/migrations/`. Never alter the DB through the dashboard, the SQL editor, or ad-hoc scripts. If a migration is wrong, write a new migration that fixes it.

> Origin: Foundational. Replaces legacy `P0 Rule #4`.

---

## RULE-COMPUTED-STATUS

`STATUS.md` is auto-generated by the post-commit hook (HC-1, AI-Native Ops). **Never edit it manually.** Reality lives in Mission Control tasks. If the status is wrong, fix the underlying task via `update_task` — do not edit `STATUS.md`.

At session start, agents must:
1. `list_tasks(status="in_progress")` to see what's actually live
2. Trust MC over any cached status file

> Origin: AI-Native Ops Phase B. Replaces legacy `P0 Rule #5` (which said "call generate_status at session start" — that's now a hook, not an agent action).

## RULE-LANGUAGE-CONTRACT

**Two languages, two layers.**

| Layer | Language |
|---|---|
| Conversation with humans (CEO, partner, contractor) | The language the human used. CEO speaks Russian → reply in Russian. Partner speaks Arabic → reply in Arabic. |
| Storage (DB, MC tasks, comments, sprint names, tags, code, code comments, commit messages, PR descriptions, `.md` specs, `AGENT.md`, schemas, migrations, frontend strings if internal) | **English only.** No exceptions. |

**Translation is the receiving agent's job.** When the CEO says "сделай Х":
1. Translate the intent to English **before** calling `emit_business_task`
2. Write the task title and description in English
3. Reply to the CEO in Russian: "Записал, id=…"

The translation happens **at the boundary**, not "later when we have time."

**Permitted exception:** the `notes` field of a task and comments **may** contain a verbatim quote from the CEO in the original language, in quotes, when tone or nuance matters. Example: `notes: "CEO said verbatim: «не трогай менеджера, он мне не нравится»"`. The surrounding text stays English.

**Why two layers:** the system has multiple human users who speak different languages. The DB and code are read by all of them (and by future agents). Storing user language in the DB creates a Tower of Babel.

> Origin: 2026-04-07. Drift detected — 1 task in 15 was russian-titled, 1 was hybrid. CEO formalized the rule before drift compounded.

## RULE-DATA-INTEGRITY

- **Naming Law:** All product codes are `ALL_CAPS_WITH_UNDERSCORES` (e.g., `RAW_CARROT`).
- **RLS Security:** Always validate row-level security policies. Assume every client request is hostile. Validate JWTs in RPCs.
- **Date Integrity:** see `engineering-rules.md` → `RULE-TXN-DATE-INTEGRITY`.

## RULE-SOCRATIC-GATE

For any new feature, DB migration, or architectural change — **stop before generating code**. Ask the CEO 2–3 specific questions covering:

1. **Scalability** — does this scale beyond the obvious case? (e.g., recursive BOM queries)
2. **SSoT alignment** — does this conflict with SYRVE or break the Supabase contract?
3. **Security** — what happens if a hostile client sends this?

Only proceed after the CEO answers.

> Origin: Foundational. Many incidents traced to "I assumed you wanted X" without asking.

---

## RULE-BACKLOG-FIRST

If, while working on task A, an agent discovers missing functionality B or has an idea for another module — **do not start B**.

The agent **must**:
1. Formulate B (what, why, expected outcome)
2. Assess priority (critical / high / medium / low — see `DISPATCH_RULES.md`)
3. Log it: `emit_business_task` with `status: inbox`, plus a one-liner in session-log
4. Report to CEO: "Discovered [B], logged as [priority]. Continue with A?"
5. **Wait for confirmation** before switching

This preserves focus, saves tokens, and prevents scope creep mid-session.

> Origin: Foundational. Without this rule, sessions drift.

---

## RULE-SELF-HEAL-TOOLING

If the same workaround, retry, or fallback fires **2+ times in a session** (or is observed across multiple sessions), the agent **must not** silently keep retrying. Instead:

1. **Diagnose** the root cause (schema mismatch, missing feature, wrong validation, etc.)
2. **Estimate effort** — is the fix under ~15 minutes and within the agent's tooling scope?
3. If **yes** → propose the fix to CEO, implement after approval
4. If **no** → create an MC task with tag `dx-friction`, domain `tech`, describe the symptom, frequency, and proposed fix

**Rationale:** Silent retries cost tokens, add latency, and erode trust. A 5-minute infra fix that eliminates 30 seconds of waste per session pays for itself in 10 sessions.

**Scope:** Applies to MCP servers, tooling, scripts, and DX infrastructure. Does **not** apply to transient network errors or rate limits.

> Origin: 2026-04-11. Observed pattern: `get_task` prefix lookup failed every session for weeks; no agent flagged or fixed it. CEO escalated.

---

## RULE-COMPOUND-ENGINEERING

If the CEO corrects a mistake, the agent **must** update the relevant file in `docs/constitution/`, `docs/domain/`, or `docs/projects/` so the same mistake never happens again. A correction without a doc update is a wasted lesson.

> Origin: Foundational. The whole constitution exists because of this rule.

---

## RULE-PLAN-BEFORE-BUILD

For any task requiring 3 or more steps, or involving architectural decisions:
1. Write a plan FIRST (in MC comment, spec file, or session-log)
2. Get confirmation from CEO (or COO for autonomous lane tasks)
3. Only then begin implementation

If something goes wrong mid-execution → STOP and re-plan immediately. Do not push through a broken approach hoping it will work out.

Simple, obvious fixes (typo, one-line change, config update) are exempt.

> Origin: 2026-04-09. Inspired by Boris Cherny's "Plan Mode Default" pattern. Complements RULE-SOCRATIC-GATE (which covers new features/migrations specifically). This rule covers the general case: any non-trivial work.
